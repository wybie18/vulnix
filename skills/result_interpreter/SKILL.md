# Result Interpreter Skill

This skill enforces a rigid analytical pipeline to sanitize, correlate, and logically interpret raw telemetry outputs generated by previous sub-agents (SAST, DAST, Fuzz, etc.) before final report integration.

## When to Activate
• Analyzing and correlating findings from multiple testing agents (Scanner outputs → Validated Vulnerabilities)
• Handling explicit False Positive downgrading through contextual flags
• Re-mapping uncontextualized scanner severities to standard CVSS v3.1 arrays

## Deduplication & Correlation Checklist

### 1. Cross-Scanner Triangulation
Single vulnerabilities often generate multiple alerts across different tool boundaries (e.g., SAST flags missing sanitization; DAST flags an alert payload).

#### ✅ ALWAYS Verify This
- Check timing and targeted endpoint parameters across all raw findings.
- Aggregate matching file lines (SAST) and HTTP Requests (DAST) into a *single* vulnerability object.
- Apply root-cause clustering: 50 different XSS payloads reflecting on `/search?q=` are merged into **one** finding mapping to the `/search` endpoint.

#### Verification Steps
- [ ] Overlapping URLs or local AST paths grouped.
- [ ] The aggregated vulnerability explicitly retains both static source lines and dynamic PoC evidence.
- [ ] [Failure Patterns](./failure_patterns.yaml) evaluated to detect logical scan blocks masking as vulnerabilities.

### 2. False Positive (FP) Nullification
All theoretical vulnerabilities must be heavily constrained by the application context.

#### ❌ NEVER Accept This
- Passing through a scanner alerting on "Missing HTTP Security Headers" (like `X-Frame-Options`) on an API endpoint that only returns `application/json`.
- Treating 403 Forbidden / 500 Internal Server Error responses generated inherently by Web Application Firewalls (WAFs) as successful exploits simply because the payload was in the URI.
- Classifying exposed developer environments in strictly isolated Docker test containers as a Critical production finding without checking deployment scope.

#### Verification Steps
- [ ] Contextual checks strictly applied to downgrades (Test/Mock setups).
- [ ] Evaluated generic Server Errors against specific stack-trace information leaks—generic errors aren't vulnerabilities unless they bleed sensitive internals.

### 3. Risk Contextualization (CVSS Mapping)
Severities must align with true business impacts mapping.

#### ✅ ALWAYS Verify This
- Base score adjustments account for authentication. (e.g., XSS inside an internal authenticated admin panel is objectively different from Unauthenticated Reflected XSS).
- Map all findings using standardized taxonomy to a precise CWE identifier.

#### Verification Steps
- [ ] Severity explicitly assigned: `Low`, `Medium`, `High`, `Critical`.
- [ ] Justification block provided specifying why a raw scanner severity was escalated or demoted.

## Analysis & Reporting Guide

### Correlation Integrity
When consolidating findings:
- Preserve the *strongest* piece of evidence. If SAST found the flaw but DAST proved it, the DAST PoC should be the primary evidence, supported by the SAST source line.

## Rules of Engagement
- **Data Preservation**: NEVER delete or strip essential raw scan data; purely filter it for downstream integration.
- **Reporting Fidelity**: NEVER hallucinate risk logic. If the provided diagnostic logs lack specific details, output the data as "Requires Manual Triage" rather than inferring exploitability.
